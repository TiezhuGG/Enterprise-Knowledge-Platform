# ADR-0001 技术栈决策

---

# 一、背景（Context）

Enterprise AI Knowledge Platform（以下简称 EAKP）是一套企业级 AI 知识平台。

平台具有以下特点：

- 企业后台管理系统
- AI 原生应用
- 大规模文档管理
- OCR
- RAG
- GraphRAG
- Workflow
- Agent
- 多租户
- 企业级权限体系

因此技术栈必须满足：

- 企业级稳定性
- 长期维护能力
- AI 代码生成质量高
- 社区活跃
- 学习成本合理
- 易于部署
- 易于扩展

本 ADR 用于冻结整个项目的技术栈。

未经新的 ADR，不允许修改。

---

# 二、设计目标（Goals）

本项目技术栈需要满足以下目标：

## G1 企业级

适用于中大型企业。

支持长期维护。

---

## G2 AI Friendly

AI 能够稳定生成代码。

生成质量高。

上下文统一。

---

## G3 全栈 TypeScript

前后端统一语言。

减少认知成本。

共享类型。

共享 DTO。

共享 Validation。

---

## G4 模块化

支持 DDD。

支持 Monorepo。

支持未来微服务拆分。

---

## G5 云原生

支持：

Docker

Kubernetes

CI/CD

---

# 三、技术选型原则

所有技术必须符合：

✔ 长期维护

✔ 社区成熟

✔ 企业应用广泛

✔ 文档完善

✔ AI 擅长生成

✔ 易于替换

禁止为了新技术而使用新技术。

---

# 四、候选方案分析

## 4.1 后端框架

方案：

A.

Spring Boot

优点：

成熟。

企业广泛使用。

缺点：

Java 开发效率较低。

AI 生成质量一般。

---

B.

FastAPI

优点：

简单。

AI 支持好。

缺点：

大型项目约束较弱。

DDD 支持一般。

---

C.

NestJS

优点：

模块化。

DDD 支持优秀。

TypeScript。

AI 生成质量高。

社区成熟。

最终选择：

✅ NestJS

---

## 4.2 前端框架

候选：

Vue3

React

Angular

最终选择：

✅ Vue3

原因：

后台管理系统成熟。

生态完善。

学习成本低。

与 Naive UI 配合优秀。

---

## 4.3 UI 框架

候选：

Element Plus

Naive UI

Ant Design Vue

最终：

✅ Naive UI

原因：

TypeScript 支持最好。

现代化。

主题能力强。

组件质量高。

---

## 4.4 ORM

候选：

TypeORM

Prisma

Drizzle

最终：

✅ Prisma

原因：

类型安全。

Migration 清晰。

AI 生成代码质量最高。

---

## 4.5 数据库

候选：

MySQL

PostgreSQL

MariaDB

最终：

✅ PostgreSQL

原因：

JSONB

全文检索

PGVector

生态完善。

---

## 4.6 向量数据库

候选：

Milvus

Qdrant

Weaviate

PGVector

最终：

✅ PGVector

原因：

无需维护第二套数据库。

适合企业项目。

降低复杂度。

未来可升级。

---

## 4.7 全文检索

最终：

✅ Elasticsearch

原因：

成熟。

企业应用广泛。

中文分词支持完善。

Hybrid Search 必备。

---

## 4.8 图数据库

最终：

✅ Neo4j

原因：

GraphRAG。

知识图谱。

关系查询能力优秀。

---

## 4.9 缓存

最终：

✅ Redis

用于：

缓存

Session

分布式锁

限流

消息

---

## 4.10 消息队列

最终：

✅ BullMQ

原因：

Node 原生。

Redis 支持。

部署简单。

适合异步任务。

---

## 4.11 文件存储

最终：

✅ MinIO

用于：

附件

图片

PDF

OCR 文件

---

## 4.12 AI 框架

最终：

✅ LangChain（作为适配层）

说明：

LangChain 不承担业务逻辑。

仅负责：

LLM

Embedding

Tool Calling

Memory

Prompt

未来允许替换。

---

## 4.13 LLM

采用 OpenAI Compatible。

支持：

OpenAI

DeepSeek

Qwen

GLM

Claude（适配）

Gemini（适配）

不绑定具体厂商。

# 六、Monorepo 结构

采用 Monorepo。

```

apps/
admin/
server/
worker/

packages/
shared/
sdk/
ui/

```

禁止：

Frontend 单独仓库。

Backend 单独仓库。

---

# 七、为什么不用微服务

当前版本：

采用模块化单体（Modular Monolith）。

原因：

开发效率高。

维护简单。

事务一致性好。

未来：

支持拆分微服务。

---

# 八、为什么不用 Milvus

原因：

增加维护成本。

需要独立部署。

对于企业知识库：

PGVector 已足够。

未来：

百万以上向量。

可新增：

ADR

升级：

Milvus。

---

# 九、影响（Consequences）

采用该技术栈后：

优势：

统一 TypeScript。

AI 生成质量高。

开发效率高。

维护成本低。

企业可落地。

缺点：

NestJS 学习成本略高。

GraphRAG 增加 Neo4j 运维。

---

# 十、禁止事项

未经 ADR：

禁止：

更换 ORM。

更换数据库。

更换框架。

增加基础设施。

---

# 十一、Review

Review 时间：

每个 Milestone Review 一次。

只有以下情况允许新增 ADR：

技术停止维护。

重大安全问题。

性能瓶颈。

业务重大变化。

否则：

保持稳定。
